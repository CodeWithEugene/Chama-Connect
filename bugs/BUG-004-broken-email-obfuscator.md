## Status update — 2026-04-20 (fixed)

Re-captured on `recon/artifacts/2026-04-20T09-40-50-508Z/html/public_contact__chamaconnect_io_contact.html`. The Email card now renders a working mailto anchor:

```html
<a class="text-green-600 font-semibold block mt-3" href="mailto:muiaalimited@gmail.com">muiaalimited@gmail.com</a>
```

`grep '\[email protected\]'` on the page returns **0 matches**. No `__cf_email__` Cloudflare obfuscator residue either. The original symptom is resolved on the live site.

**Caveat:** the site now uses `muiaalimited@gmail.com`, which does not match the hackathon brief's `support@muiaa.com` (the canonical address suggested in BUG-003's single-source-of-truth proposal). If the product is meant to migrate onto its own domain at some point, this just moves the inconsistency; the contact-info-canonicalization fix in BUG-003 should still happen.

## Original evidence (historical)

From `/tmp/cc_contact.html`:

```
Email
Reach us anytime for inquiries or support.
[email protected]
```

The string `[email protected]` was rendered verbatim in the Email card on `/contact`. Users saw the WordPress/Cloudflare placeholder instead of an actual address.

## User impact

A user who wants to email support sees a placeholder, shrugs, and bounces. For a financial product, making support impossible to reach is a severe trust/retention problem.

## Root cause

Two likely causes, in decreasing probability:

1. **Cloudflare email obfuscation** was enabled on the site but the JS that replaces `/cdn-cgi/l/email-protection` links never fired because the element was rendered as plain text, not an anchor with a `data-cfemail` attribute. The literal `[email protected]` is Cloudflare's fallback.
2. A template copy-paste from a WordPress-based mock.

## Proposed fix

1. Render the address as a proper `mailto:` anchor and let Cloudflare handle obfuscation correctly (or disable CF email obfuscation for this route):

```tsx
// app/contact/page.tsx — before
<p>[email protected]</p>

// after
import { siteConfig } from "@/config/site";
<a
  href={`mailto:${siteConfig.email}`}
  className="font-semibold text-primary hover:underline"
>
  {siteConfig.email}
</a>
```

2. If you keep CF obfuscation on, use an `<a class="__cf_email__" data-cfemail="...">` generated by Cloudflare instead of plain text.

3. Single source of truth: `siteConfig.email = "support@muiaa.com"` (from the hackathon brief) — used everywhere.

## Verification

- Visit `/contact` — the Email card should show an actual, clickable `support@muiaa.com`.
- `curl -s https://chamaconnect.io/contact | grep -c '\[email protected\]'` should return `0`.
- Clicking the link opens the user's mail client to a blank email to `support@muiaa.com`.
